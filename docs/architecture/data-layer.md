# Data Layer

Two databases, one direction of truth.

## Firestore — source of truth

All human-facing endpoints (`/api/admin/*`, `/api/org/*`, `/api/mobile/*`,
client SDK reads) read and write Firestore.

- Server access: `import { adminDb } from '@/lib/firebaseAdmin'`
- Client access: `import { db } from '@/lib/firebaseClient'`
- Rules: `src/firebase/firestore.rules`
- Indexes: `firestore.indexes.json`
- 30 active collections (organizations, locations, users, tasks,
  taskCompletions, sessions, media, videos, etc.).

## PostgreSQL — read-only replica

Powers the Robot Intelligence API (`/api/robot/v1/*`) only. Never written
to from application code.

- Driver: `@neondatabase/serverless` (HTTP + WebSocket, Neon-specific)
- Wrapper: `src/lib/db/postgres.ts` exposes tagged template `sql\`...\``
  and parameterized `sql.query(text, params)`
- Schema: hand-written `database/schema.sql` (canonical for fresh
  bootstrap), versioned in Drizzle from Phase F onward.
- Sync: `/api/cron/sync-sql` runs daily (defined in `vercel.json`),
  driven by `src/services/firebase-to-sql-sync.service.ts`. Pulls from
  Firestore, normalizes, upserts into Postgres.

## Drizzle Kit (schema-only)

Introduced in Phase F of repo cleanup. Owns schema definitions
(`src/lib/db/schema.ts`) generated by `drizzle-kit pull` against the
live replica. Future schema changes use `drizzle-kit generate` to emit
versioned SQL migrations under `drizzle/`.

Repository code keeps using raw SQL via the `sql` template — Drizzle is
**not** an ORM in this codebase, only a migration manager.

## Boundary rules

- Robot endpoints: PostgreSQL only. Use `sql` from `@/lib/db/postgres`.
- Everything else: Firestore only. Use `adminDb` or `db`.
- Never write to PostgreSQL from application code outside the cron sync.
- Never read PostgreSQL from human-facing endpoints — replica lag is
  ~24h.

## Future GCP migration

- Firestore stays — Firebase Firestore *is* GCP Firestore. SDK init
  changes from service-account JSON to ADC; no data migration.
- PostgreSQL → Cloud SQL Postgres. `@neondatabase/serverless` swaps for
  `pg` Pool via Cloud SQL Auth Proxy. Public API of `postgres.ts`
  stays identical.
- Drizzle baseline lets cutover bootstrap Cloud SQL with a versioned
  migration history instead of one-shot `schema.sql`.
